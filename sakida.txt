# AliExpress-Only Business Gap Map

Audit date: 2026-03-21

## Verdict

AliExpress-only is not commercially sufficient today, but it is also not truthfully exhausted.

The current failure pattern still mixes real supplier-side scarcity with unresolved implementation and operational gaps. Under the current business policy, the correct conclusion is:

- do not switch suppliers yet as the main recommendation
- fix the unresolved AliExpress-only truth and execution gaps first
- re-test first-profit readiness only after the current AliExpress-only path is genuinely exercised end-to-end

## Usable AliExpress Capabilities Today

| Capability | Current state | What is genuinely strong | What is weak |
| --- | --- | --- | --- |
| Affiliate discovery | PARTIAL | Search and product-detail logic exist; good for candidate mining | Discovery still does not produce validated-ready output |
| Dropshipping product validation | STRONG BUT INCOMPLETE | Real `getProductInfo`, SKU parsing, destination-sensitive checks | Parameter mismatches and incomplete catalog truth still cause failures |
| Preventive supplier audit | STRONG BUT INCOMPLETE | Original supplier plus ranked alternatives; rating/order filters; keyword and similarity gates | No live first-safe candidate produced |
| Fallback ranking | PARTIAL | Ranked alternative supplier selection is real | It has not yet delivered one commercially usable fallback in production |
| Alternative-product fallback | PARTIAL | Code exists for supplier alternatives and URL substitution | No commercial proof that it rescued a real order safely |
| Stock checks | PARTIAL | SKU-level stock checks are real | Catalog lacks persisted SKU coverage almost everywhere |
| Shipping checks | PARTIAL | API-based destination shipping logic is real | Live failures still show destination parameter and shipping truth issues |
| Cost extraction | PARTIAL | Supplier unit cost and shipping cost extraction exist | Persisted total-cost coverage is near zero |
| Product identity / SKU resolution | PARTIAL | Strong selection logic exists | Live failures still show `SKU_NOT_EXIST` and missing SKU persistence |
| Destination compatibility | PARTIAL | Destination-context resolution is strict in the preventive path | Most products do not carry destination-safe persisted context |
| Purchase execution | PARTIAL | Dropshipping API `placeOrder` path exists | Supplier-side payment closure is not proven |
| Tracking retrieval | PARTIAL | Tracking API surface exists | No real successful-operation record proves it closed a profitable lifecycle |

### Supplier-side rejection evidence carried into this audit

Previous documented recovery evidence cited:

- `no_stock_for_destination`
- `margin_invalid`

Current live evidence still supports those concerns indirectly:

### Meaning of that evidence

AliExpress-only has clearly been stressed enough to prove that it is hard.

It has not yet been stressed enough to prove that it is impossible under the current rules, because the stack still has unresolved implementation and operational gaps that contaminate the result:

- incomplete persisted product truth
- unstable marketplace auth on the lead path
- no proven supplier-side PayPal payment closure
- legacy/test artifacts mixed into post-sale evidence

## Strengths Inside the AliExpress-Only Path

1. The preventive supplier audit is commercially serious.

- `NOT YET EXHAUSTED` strategically

# Prioritized Improvement Plan to Real Profit

Audit date: 2026-03-21

## P0: Absolute Blockers to the First Profitable Operation

### P0.1 Recover the first truthful `VALIDATED_READY`

- Exact issue: the validated catalog is empty and product truth persistence is almost entirely missing.
- Why it matters: no validated-ready item means no safe publish, no safe order, and no real profit path.
- Affected subsystem: preventive supplier validation, pre-publish validator, product persistence.
- Recommended fix: run a narrow AliExpress-only recovery program that persists `targetCountry`, `shippingCost`, `importTax`, `totalCost`, and `aliexpressSku` only through the strict preventive path.
- Expected business impact: creates the first candidate that can legally and economically move into commerce.
- Validation proof required: DB shows `VALIDATED_READY >= 1` with all required fields populated and audit metadata present.

### P0.2 Remove or hard-block stale publish surfaces

- Exact issue: at least one publish surface still operates on stale semantics like `status = publishable`.
- Why it matters: it creates false readiness and can produce unsafe listing attempts.
- Affected subsystem: marketplace publish orchestration.
- Recommended fix: make `VALIDATED_READY` the only acceptable publish input and fail closed everywhere else.
- Expected business impact: prevents unsafe listings and keeps evidence trustworthy.
- Validation proof required: any non-validated-ready publish attempt hard-fails with a clear message; one validated-ready publish succeeds.

### P0.3 Repair eBay lead-path auth and order-sync readiness

- Exact issue: live production logs show repeated eBay refresh failures with HTTP `400`.
- Why it matters: even with safe inventory, the lead marketplace cannot sustain real order flow.
- Affected subsystem: eBay connector, OAuth token refresh, credential persistence.
- Recommended fix: reconcile the live production credential set for the intended operator account, then re-run bounded readiness checks and live order-sync validation.
- Expected business impact: restores the strongest available marketplace route.
- Validation proof required: no refresh failures in recent logs, healthy production snapshot, and successful bounded probe.

### P0.4 Repair supplier request correctness for real Dropshipping execution

- Exact issue: recent failures show missing required parameters and stale SKU/product references.
- Why it matters: automatic purchase cannot be trusted.
- Affected subsystem: order fulfillment, AliExpress Dropshipping request construction.
- Recommended fix: normalize request payload generation for destination, quantity, SKU, and product identity; reject incomplete payloads before hitting AliExpress.
- Expected business impact: converts supplier purchase from fragile to testable.
- Validation proof required: one real non-test order reaches a real supplier order without `MissingParameter`, `SKU_NOT_EXIST`, or `PRODUCT_NOT_EXIST`.

### P0.5 Close the supplier-side PayPal payment gap

- Exact issue: the intended AliExpress purchase-via-PayPal step is not proven or automated.
- Why it matters: without supplier payment closure, the business loop is incomplete.
- Affected subsystem: AliExpress purchase path, PayPal integration, supplier payment orchestration.
- Recommended fix: implement and prove the exact supplier-side payment completion step, or explicitly redesign the business policy if Dropshipping API cannot support PayPal settlement.
- Expected business impact: turns “order created” into “supplier actually paid”.
- Validation proof required: one real supplier purchase shows both supplier order creation and payment completion.
