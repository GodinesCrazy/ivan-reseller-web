# ML Image Real Case Remediation

> Date: 2026-04-02 | Product: Cat stand (AliExpress) | Marketplace: Mercado Libre Chile

---

## The Problem

The product image pack was being rejected by ML's automated moderation despite passing all internal validation checks. The listing was either:
- Published but demoted in search rankings (`poor_quality_thumbnail` tag)
- Or rejected at upload with "No cumple con los requisitos de fotografía"

## Root Cause

Internal investigation revealed that `autoGenerateSimpleProcessedPack()` was falling through to **Phase 2** (soft background neutralization) for this product's images, and Phase 2 was producing a **gray background** (#DCDCDC = RGB 220,220,220) — not a white background.

### Why Phase 1 was failing

AliExpress images of this product have a warm-gray background (typical AliExpress "studio gray" ~RGB 194,180,163). The border-statistics isolation algorithm:

1. Samples border pixels → computes background mean ~(194, 180, 163)
2. Classifies pixels close to this mean as background
3. The product has some dark/mid-tone sections but also lighter transitions

For 3 of 4 segmentation variants, the binary mask either:
- Included too much background (warm-gray confusion) → composition quality gate rejects (>5% warm-gray)
- Or isolated the product but on a white canvas it appeared at ~60% area coverage, which was fine

The real issue was the quality gate (`>95% white` or `>5% warm-gray`) was rejecting valid Phase 1 outputs for some images, forcing Phase 2.

### Why Phase 2 was wrong

```
SOFT_GRAY = 220   // was #DCDCDC — NOT white
grayBg = { r: 220, g: 220, b: 220 }
```

Output: product on **gray canvas** → 0% pure-white pixels → fails ML `white_background` check.

### Why this wasn't caught internally

`portadaGateBypass: true` was set unconditionally in the manifest for ALL `autoGenerateSimpleProcessedPack` outputs. This caused `inspectAsset()` to skip `evaluateMlPortadaStrictAndNaturalGateFromBuffer()`, which includes the white background check.

Result: the internal pipeline reported `approvalState: 'approved'` even for a gray-background image.

## The Fix Applied

### Change 1: Phase 2 gray → white (critical fix)

**File:** `backend/src/services/mercadolibre-image-remediation.service.ts`

```typescript
// BEFORE:
const SOFT_GRAY = 220; // #DCDCDC — ML rejects as non-white
const grayBg = { r: SOFT_GRAY, g: SOFT_GRAY, b: SOFT_GRAY };

// AFTER:
const SOFT_WHITE = 255; // #FFFFFF — ML-compliant white background
const whiteBg = { r: SOFT_WHITE, g: SOFT_WHITE, b: SOFT_WHITE };
```

All pixel replacement and canvas extension now use white instead of gray.

### Change 2: Quality gate added to Phase 2

Previously, any Phase 2 candidate was accepted (visibility score just needed to be > -1). Now:

```typescript
// New: reject if product is invisible (all near-white)
if (nearWhitePct > 0.95) {
  logger.warn('[autoGenerateSimpleProcessedPack] soft-white candidate >95% near-white — product invisible, skipping');
} else if (visibilityScore > bestVisibilityScore) {
  // accept this candidate
}
```

### Change 3: Phase tracking in manifest notes

```typescript
notes: `generated by internal 80pct-fit white-bg remediation | phase=${coverPhaseLabel} | bg=white_255 | ml_white_background_compliant`
```

### Change 4: Stricter white background thresholds

**File:** `ml-portada-visual-compliance.service.ts`

```typescript
// BEFORE:
const WHITE_NEAR_DOMINANCE_MIN = 0.58;
const WHITE_PURE_DOMINANCE_MIN = 0.28;

// AFTER:
const WHITE_NEAR_DOMINANCE_MIN = 0.62;
const WHITE_PURE_DOMINANCE_MIN = 0.40;
```

Gray (#DCDCDC) now definitively fails even if `portadaGateBypass` is not set.

## Expected New Behaviour for This Product

With the fix:

1. **Phase 1** will attempt isolation for all available source images × 4 variants.
   - If any produces a clean product on white with 35–91% coverage → accepted ✓
   - The quality gate (>5% warm-gray) still applies but warm-gray bleed now more clearly rejected

2. **Phase 2** (if Phase 1 still fails) will:
   - Replace background pixels with pure white (#FFFFFF)
   - Check product visibility (reject if >95% near-white)
   - Produce a white-background image that passes ML's `white_background` check

3. **Manifest** notes record `phase=phase2_soft_bg_neutralization_white | bg=white_255`

4. **Internal V1 gate** (if bypass is ever removed): white dominance thresholds (0.62/0.40) will correctly approve white-background outputs and reject any gray-background output.

## Residual Risk

1. **Phase 3 (absolute fallback):** If both Phase 1 and Phase 2 fail (product is near-white and gets lost in both approaches), Phase 3 uses `flatten + extend` which preserves the source image's original background. For a gray-background AliExpress image, Phase 3 would still produce a gray background. **Mitigation:** Phase 3 should only fire for images that are already predominantly white/near-white (where Phase 2 correctly rejects them as >95% near-white), in which case the original image may actually have a close-to-white background anyway.

2. **Product with very light coloring:** If the actual product is cream/off-white, it may still get confused with the white background. In this case, human review of the image pack is recommended before publishing.

3. **portadaGateBypass: true scope:** The bypass still disables the strict+natural gate (text/collage detection) for Phase 1 outputs. This is intentional (clean isolation outputs are trusted), but Phase 2 outputs should arguably NOT bypass text detection. If Phase 2 output ever contains a source image with text, the strict gate would not catch it. **Mitigation:** Phase 2 only applies soft pixel-level background neutralization — it cannot introduce new text that wasn't in the original. The original source images have already been screened by `auditMercadoLibreChileImagePolicy()`.
